Sure, let's look at an example to understand how Cross-Site Scripting (XSS) can occur. Consider a simple web application that has a feature allowing users to post comments. The server displays these comments on a webpage without any filtering or escaping of HTML characters.

Example of a Stored XSS Attack:

  1. User Input: Suppose an attacker writes a comment on the website, embedding a malicious script within it. The comment might look like this:
Great post! <script>alert('Hacked!');</script>
  1. Storing the Comment: The web application stores this comment in its database without any sanitization.
  2. Displaying the Comment: Later, when another user visits the page where comments are displayed, the web application retrieves the comment and includes it in the HTML of the page.
  3. Execution of the Script: When the user's browser renders the HTML, the embedded <script> tag is treated as a legitimate script and executed. The user sees an alert box with the message 'Hacked!'.

This is a classic example of a stored XSS attack, where the malicious script is stored on the server and executed by any user who views the infected page.
